# 02. 需求与范围

## 1. 功能需求

### 1.1 核心功能
工具只提供两个核心命令：

1. `pack`
   - 输入：目录
   - 输出：单文件归档

2. `unpack`
   - 输入：单文件归档
   - 输出：目录树

### 1.2 压缩能力
压缩时支持：

- 指定压缩算法
- 指定线程数
- 指定 bundle 参数
- 指定完整性模式
- 指定元数据保留策略

### 1.3 解压能力
解压时支持：

- 从协议头读取 codec
- 从协议头读取建议并发度
- 从顺序流中解析 header / manifest / frame
- 将数据恢复为目录树
- 可通过 CLI 覆盖线程数和恢复策略

## 2. 语义范围

### 2.1 与 tar 的对齐范围
SFA 的语义目标不是字节兼容 tar，而是对齐 tar 在 Unix 系统上的常见文件树表达能力。

v1 的实现范围如下：

| 类型 | 协议层 | v1 实现 | 说明 |
|---|---:|---:|---|
| regular file | 支持 | 支持 | 核心路径 |
| directory | 支持 | 支持 | 核心路径 |
| symlink | 支持 | 支持 | Unix 必需 |
| hardlink | 支持 | 支持 | 通过 `(dev, ino)` 识别 |
| fifo | 预留 | 可延后 | 默认不建议开启 |
| char device | 预留 | 可延后 | 涉及权限与安全 |
| block device | 预留 | 可延后 | 涉及权限与安全 |
| xattrs | 预留 | 次阶段 | 通过扩展元数据区 |
| ACL | 预留 | 次阶段 | 通过扩展元数据区 |
| uname/gname | 预留 | 次阶段 | 可选显示信息 |

### 2.2 v1 必须恢复的元数据
- type
- path
- mode
- uid
- gid
- mtime
- size
- symlink target
- hardlink target

### 2.3 v1 恢复策略约束
- `uid/gid` 恢复应由开关控制
- 非 root 用户默认不强制 chown
- 目录 mtime 应在所有子项恢复完成后再设置
- hardlink 必须在主文件存在后创建
- symlink 不得参与中间路径逃逸

## 3. 非功能需求

## 3.1 性能
### 目标
- 海量小文件场景吞吐优先
- 多核机器上具备良好并行扩展
- 高并发下内存可控

### 允许的取舍
- 牺牲一部分压缩比
- 增加一定内存缓冲
- 增加预扫描成本
- 采用更复杂的协议结构

## 3.2 流式
协议必须满足：

- header 可先解析
- manifest 可在数据段前完整获取
- frame 可顺序解码
- 正常解压不依赖 trailer
- 正常解压不依赖 seek

## 3.3 稳定性
- 损坏归档应尽早失败
- 截断流应快速报错
- 对非法路径、恶意输入有防护
- 对版本不兼容有清晰错误

## 3.4 可扩展性
- 新 codec 可扩展
- 元数据能力可扩展
- 完整性算法可扩展
- 可以逐步增加 async API / 服务化能力

## 4. 约束条件

### 4.1 平台约束
- v1 仅针对 Unix 系统
- 不试图解决 Windows 权限、路径、链接模型差异

### 4.2 协议约束
- 数据协议必须统一支持本地文件流和 HTTP 网络流
- 不能要求网络流具备随机访问能力
- 不能要求先落临时文件再解压

### 4.3 工程约束
- Rust workspace 组织
- CLI 与 library 共存
- 协议规范先于实现冻结
- benchmark 必须纳入仓库

## 5. 对标方式

### 5.1 对标对象
统一对标：

- `tar + lz4`
- `tar + zstd`

禁止使用“不同算法之间”的交叉对比来得出结论，例如：

- 不能拿 `sfa + lz4` 去对比 `tar + zstd`
- 不能拿 `sfa + zstd` 去对比 `tar + gzip`

### 5.2 对标维度
- pack wall time
- unpack wall time
- 文件数吞吐（files/s）
- 原始数据吞吐（MiB/s）
- CPU 占用
- RSS
- 输出体积

## 6. v1 范围定义

### 6.1 Must
- header + manifest + frame 协议
- LZ4 / Zstd
- bundle 级并行
- 流式解压
- regular / directory / symlink / hardlink
- mode / uid / gid / mtime
- fast / strong 完整性
- benchmark 对标 tar

### 6.2 Should
- CLI 覆盖线程数
- 进度与统计输出
- 有界队列背压
- golden 样例
- 版本兼容测试

### 6.3 Could
- xattrs / ACL
- special files
- async unpack API
- 更细粒度 observability

### 6.4 Won't（v1 不做）
- 随机访问归档内文件
- 增量归档
- 内容去重
- 加密 / 签名
- 归档修改
- 跨平台语义统一

## 7. 关键取舍

### 7.1 为什么允许预扫描目录
因为预扫描能换来：

- manifest 前置
- 确定 bundle 规划
- 更稳定的并行调度
- 更清晰的 hardlink 处理
- 更好的 deterministic output

### 7.2 为什么不做“每文件一个压缩流”
因为那会保留大量小文件场景中的核心开销：

- 过多压缩上下文切换
- 每文件边界太密
- 并行粒度太碎
- 小对象调度成本高

### 7.3 为什么协议头中的线程数只存“建议值”
因为归档可能在高配机器上生成，在低配机器上解压。  
头部应记录：

- 压缩时使用的并发建议
- 不是解压端必须遵守的硬约束

## 8. 输出目录与扩展名建议

- 输出格式扩展名：`.sfa`
- 命令：`sfa pack` / `sfa unpack`

## 9. 范围结论

SFA v1 的边界应该非常清晰：

> **聚焦“海量小文件的高吞吐归档”这一件事。**  
> 不为了通用性牺牲核心链路，不为了功能面而稀释性能目标。
