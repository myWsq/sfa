# 01. 项目总览

## 1. 项目定位

SFA（Streaming Folder Archive）是一个面向**海量小文件**场景的高吞吐归档工具。  
它的设计目标不是“做一个通用压缩包格式”，而是解决以下典型问题：

- 数十万到数百万个小文件的打包吞吐过低
- tar 在海量小文件上存在明显的元数据和小 IO 开销
- 常见格式对严格流式并行解码支持不够友好
- 网络流输入（如 HTTP body）和本地文件输入不应有协议层差异

## 2. 目标用户

主要面向以下场景：

- 构建产物缓存与分发
- 容器层导出后的文件树打包
- 海量小源码文件、配置文件、json、日志切片归档
- CI/CD 中间产物传输
- 对吞吐优先、压缩比次优的内部工具链

## 3. 业务目标

### 3.1 主目标
在海量小文件场景下，相比 `tar + 同压缩算法`，获得**显著更高的打包 / 解包吞吐**。

### 3.2 次目标
- 协议天然支持流式读取
- 解压阶段不依赖随机访问
- 保持 Unix 文件树语义
- 对调用方提供简单稳定的 CLI 和库接口

### 3.3 非目标
以下能力不作为 v1 目标：

- 与 tar/zip/7z 的字节级兼容
- 优先追求最高压缩比
- 面向 Windows 的语义兼容
- 单个归档内随机读取指定文件
- 加密、签名、断点续传、增量归档

## 4. 已确认输入条件

### 4.1 已确认
- 语言：Rust
- 平台：Unix-only
- 允许压缩前全量扫描目录
- 协议支持从本地文件流、HTTP 网络流中顺序解析
- 压缩时可指定算法和线程数
- 解压从头部读取算法与建议并发度
- 一切设计以更高吞吐为最高优先级
- 可以接受更高内存占用
- 对标 tar

### 4.2 推导出的关键设计约束
由上述条件可以直接推导出以下架构约束：

1. **manifest 必须位于数据段前部**
   - 否则严格流式解压会缺少目录树与目标文件映射信息

2. **manifest 不应依赖压缩后偏移**
   - 因为压缩后的 frame 长度在压缩前未知
   - 否则必须 seek 回填

3. **数据段必须由自描述 frame 构成**
   - 才能在本地流与网络流中统一消费

4. **压缩/解压并发应建立在格式级工作单元上**
   - 即 bundle/frame 级并发
   - 而不是过度依赖某个 codec 的内部线程模型

## 5. 方案核心思想

### 5.1 tar 语义，非 tar 结构
SFA 在语义上尽量与 tar 的 Unix 文件树表达保持一致，但在字节格式上采用完全不同的结构：

- tar：按文件顺序线性追加 header + body
- SFA：`header + manifest + frame* + trailer?`

### 5.2 工作单元从“文件”升级为“bundle”
海量小文件的吞吐瓶颈通常不是纯压缩算法，而是：

- 目录遍历与元数据访问
- 大量小读写 syscall
- 频繁的压缩器上下文切换
- 小对象调度开销
- 写出顺序与回填复杂度

SFA 通过以下手段解决：

- 小文件聚合成 bundle
- bundle 作为压缩/解压的工作单元
- 文件与 bundle 的关系通过 manifest 建模
- writer 只需要按 bundle_id 顺序写出 frame

## 6. 设计原则

### 6.1 流式优先
协议设计必须满足：

- 解压器只依赖 `Read` / `AsyncRead`
- 正常解压不依赖 `Seek`
- 从 HTTP body 读取和从本地文件读取行为一致

### 6.2 吞吐优先
允许为了更高吞吐做以下取舍：

- bundle 内部打散小文件边界
- 使用更大内存缓冲
- 默认选择更偏速度的压缩算法
- 不追求极致压缩比

### 6.3 简单优先
v1 方案避免同时引入过多高级特性：

- 不做随机访问索引
- 不做去重
- 不做多算法混合 frame
- 不做复杂远程写回协议

### 6.4 可扩展优先
虽然 v1 边界收敛，但协议必须预留以下扩展位：

- xattrs / ACL
- special files
- trailer 强校验
- manifest 扩展元数据
- 新 codec
- 新完整性算法

## 7. 成功标准

### 7.1 功能成功标准
- 可完整压缩 / 解压目录树
- 可恢复 regular file / directory / symlink / hardlink
- 目录结构与文件内容正确
- mode / uid / gid / mtime 恢复策略明确且可控
- 支持本地文件与 HTTP 流顺序解压

### 7.2 性能成功标准
- 在海量小文件数据集上，相比 `tar + 同 codec`，pack/unpack 吞吐具有明显优势
- CPU 使用更均衡
- RSS 处于可控范围
- 大文件场景不明显退化

### 7.3 工程成功标准
- 协议冻结后具备 golden test
- CLI 可用
- 基准框架可重复运行
- 错误处理与日志可观测
- 具备开源发布的最小工程化能力

## 8. 项目命名建议

工作名建议采用：

- 仓库名：`sfa`
- 协议名：`SFA v1`
- CLI：`sfa`
- 归档扩展名：`.sfa`

这组命名足够简洁，也易于在 README 中传达项目定位。

## 9. 一句话总结

> SFA 是一个面向海量小文件的高吞吐流式并行归档工具：  
> **用 tar 的文件树语义，配合前置 manifest 与 bundle/frame 流水线，实现优于 tar 的端到端吞吐。**
