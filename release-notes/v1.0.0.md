# SFA v1.0.0 Release Notes

## Highlights

- Publish the first stable SFA v1 release around the frozen `format-v1` protocol and the current manifest-first Unix archive toolchain.
- Ship the stable v1 surface for regular files, directories, symlinks, hardlinks, safe sequential unpack, default `mode` / `mtime` restore, and explicit opt-in owner restoration.
- Keep `zstd` at level `-3` as the default `sfa pack` data codec and preserve the machine-readable pack / unpack stats used by benchmarks and automation.
- Include the post-`v0.3.0` unpack directory setup optimization from the current `main` candidate revision together with refreshed benchmark evidence.

## Compatibility

- No wire-format change is introduced in this release; `format-v1` remains the frozen compatibility boundary for SFA v1.
- Existing archives from the current v1 line remain readable, and explicitly selected codecs continue to work as before.
- The default output of `sfa pack` remains `zstd` data frames at level `-3` when `--codec` is omitted.
- Automation that parses unpack stats must continue to treat `wall_breakdown` as the additive wall-time view and `phase_breakdown` as overlapping diagnostic pipeline windows rather than expecting phase fields alone to sum to total duration.
- `v1.0.0` does not expand the stable metadata contract to xattrs, ACLs, special files, or non-Unix parity.

## Known Gaps

- Post-v1 Unix metadata work remains deferred, including xattrs, ACLs, special-file restore, and any broader Unix extension surface.
- SFA still does not promise fully equivalent non-Unix behavior, crates.io distribution, installer-style packaging, macOS notarization, or code signing.
- Large real-world diagnostic thread sweeps remain supporting evidence rather than formal release gates.
