# SFA v0.3.0 Release Notes

## Highlights

- Make `zstd` at level `-3` the default data codec for `sfa pack`, reducing the need to opt in explicitly to the repository's preferred compressed archive path.
- Tighten unpack setup so restore work spends less time in serial pre-work before worker scatter begins, especially on small-file workloads.
- Expand machine-readable unpack timing with additive `wall_breakdown` buckets (`setup`, `pipeline`, `finalize`) while keeping `phase_breakdown` as the diagnostic pipelined view.
- Fix the GitHub release workflow's macOS Intel runner configuration so tagged releases can build the intended archive set.

## Compatibility

- No wire-format change is introduced in this release; `format-v1` remains the frozen compatibility boundary for SFA v1.
- Existing archives remain readable, and explicitly selected codecs continue to work as before.
- The default output of `sfa pack` changes when `--codec` is omitted: new archives now use `zstd` data frames at level `-3` instead of the previous default.
- Automation that parses unpack stats should treat `wall_breakdown` as the additive wall-time view and `phase_breakdown` as overlapping diagnostic pipeline windows rather than expecting phase fields alone to sum to total duration.

## Known Gaps

- M3 Unix semantics hardening is still in progress, including deeper metadata and owner-policy verification.
- SFA still does not promise xattrs, ACLs, device file restore, non-Unix parity, or crates.io distribution.
- GitHub Releases publish archive artifacts, but macOS binaries remain unsigned and not notarized.
